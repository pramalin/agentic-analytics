# Query Guard

> **Status as of Step 5b:** this is no longer the primary enforcement
> mechanism for the live agent — the MCP gateway's postgres server connects
> as a DB-layer read-only role instead (see
> [`docs/mcp-gateway.md`](mcp-gateway.md)). `QueryGuard` and its tests
> remain in the codebase as a still-valid second layer and as the
> enforcement for `AgentTools`, the in-process alternative kept for
> reference. Everything below describes what `QueryGuard` itself does; it
> doesn't currently sit in the request path when you ask the agent a
> question through `/api/questions`.

`QueryGuard` (`application/src/main/java/com/example/agenticanalytics/datamart/QueryGuard.java`)
is the last check a SQL string passes through before it reaches the data
mart. Every query `DataMartQueryService.runQuery()` runs — whether typed by
a human or generated by the agent — goes through `QueryGuard.assertReadOnly()`
first.

## Why it exists

Once the agent (Step 5+) is writing its own SQL against `run_query`, nothing
upstream guarantees the model only ever proposes a `SELECT`. Prompt
instructions ("only write SELECT statements") shape behavior but are not a
security boundary — a model can be wrong, and a tool exposed over MCP
(Step 7) may eventually be called by clients this codebase doesn't control.
`QueryGuard` is the point where "only reads are possible" stops being a
request and starts being enforced in code.

## What it does

`assertReadOnly(String sql)`:

1. Rejects `null` or blank input.
2. Requires the statement to *start* with `SELECT` (case-insensitive,
   leading whitespace tolerated).
3. Scans the (uppercased) statement for a fixed list of disallowed
   keywords — `INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, `TRUNCATE`,
   `CREATE`, `GRANT`, `REVOKE`, `MERGE`, `CALL`, `EXEC` — and for `;`, which
   would allow statement stacking (`SELECT ...; DROP TABLE ...`).

Anything that fails either check throws `IllegalArgumentException`, which
`DataMartQueryService` does not catch — it propagates up to the caller
(currently the MCP/tool layer), so a rejected query surfaces as a tool
error the agent has to explain, not a silent no-op.

## What it deliberately does NOT do

This is a regex tripwire, not a SQL parser, and that's a conscious scope
decision, not an oversight:

- **It does not understand SQL grammar.** It cannot tell that
  `SELECT * FROM t WHERE name = 'DROP TABLE'` is a harmless literal
  string containing the word `DROP` — that query would currently be
  *rejected* (a false positive, safe direction to fail in).
- **It does not prevent expensive queries.** A syntactically valid
  `SELECT` with an unbounded cross join or no `WHERE` clause passes the
  guard fine. `DataMartQueryService` caps result rows at 200, but nothing
  here caps query *execution time* or database load.
- **It is not a substitute for database-level permissions.** The
  application's Postgres user should also be a genuinely read-only role
  (`GRANT SELECT` only, no `INSERT`/`UPDATE`/`DELETE`/DDL grants) as a
  second, independent layer. Today it is not — the same `postgres`
  superuser is used everywhere in this project for simplicity. That's
  fine for a local demo; it would not be fine to ship as-is.
- **It does not prevent data exfiltration via SELECT.** A `SELECT *` over
  a sensitive table is, by design, something this guard allows — that's
  a query-scope/tool-design concern, not something a read-only check can
  or should solve.

## Testing

- `QueryGuardTest` (`application/src/test/java/.../datamart/QueryGuardTest.java`)
  — unit tests, no Spring context, no database. Covers: simple `SELECT`,
  `SELECT` with `JOIN`/`WHERE`/`GROUP BY`, each disallowed keyword,
  statement-stacking via `;`, blank input, `null` input.
- `DataMartQueryServiceIT` includes one end-to-end check
  (`runQueryRejectsWriteStatements`) confirming the guard is actually wired
  into the real JDBC path, not just unit-tested in isolation.

## Possible future hardening

Not implemented yet; listed here so the gap is visible rather than silent:

- Replace the regex approach with a real SQL parser (e.g. JSqlParser) that
  validates statement *type* structurally rather than by keyword-scanning
  the raw string — would close both the false-negative risk of a keyword
  hidden inside a quoted identifier or comment, and the false-positive risk
  described above (`DROP` inside a string literal).
- Add a query-cost/timeout guard (e.g. `SET statement_timeout` per
  connection, or an `EXPLAIN`-based cost check before execution).
- Provision a genuinely read-only Postgres role for the application user,
  independent of this application-layer check.
